# Product Requirements: WHMCS PTR / rDNS and Outgoing Port 25 Approval Module

## Product Name

Bisup WHMCS PTR & Port 25 Approval Module

## Module Type

WHMCS Addon Module

The module runs inside WHMCS and provides:

- Client-side request form
- Admin-side moderation panel
- KYC document upload
- PTR / rDNS request management
- Outgoing Port 25 approval management
- Staff approval workflow
- Audit logs
- Ticket/email notification support

## Goal

Create a controlled approval system inside WHMCS before enabling PTR/rDNS records, outgoing SMTP Port 25, or mail server sending permission.

The module is intended to reduce abuse risk, protect IP reputation, and ensure every request has KYC, moderation, and audit history.

## Target Users

### Client

WHMCS clients who need PTR/rDNS or outgoing Port 25 enabled for a VPS, dedicated server, cloud server, or mail server.

### Support Staff

Staff who review submitted requests and verify client details.

### Admin

Senior staff who can approve, reject, enable, suspend, or flag requests.

## Client Area Features

### Request Page

Client area page:

```text
Client Area > Services > PTR / Port 25 Request
```

Clients can submit a request for:

- PTR / rDNS only
- Outgoing Port 25 only
- Both PTR / rDNS and Port 25

### Service Selection

The client must select an active WHMCS service. Eligible products are configured by the module administrator.

The system should not allow requests for cancelled, suspended, terminated, or unpaid services.

### Required Client Fields

- Service ID
- Server IP address
- Requested PTR hostname
- Mail sending domain
- Reason for request
- Type of email usage
- Estimated daily email volume
- Mail server software
- Business/organization name
- Website URL
- Contact person name
- Contact number

### Mail Usage Type

The client must select one:

- Transactional emails
- Business emails
- Hosting client emails
- Application/system emails
- Newsletter with consent
- Other

Bulk unsolicited email, spam, phishing, spoofing, or illegal email usage must be rejected.

### DNS Readiness Fields

The client must confirm:

- Domain has a valid A record
- SPF is configured
- DKIM is configured or planned
- DMARC is configured or planned
- Mail hostname is valid
- PTR hostname belongs to the client's domain

### KYC Upload

Client must upload KYC documents before submitting.

For individual clients:

- Citizenship / National ID / Passport
- Address proof, if required

For company clients:

- Company registration certificate
- PAN/VAT document, if available
- Authorized person ID

Supported file types:

- PDF
- JPG
- PNG
- JPEG

Maximum file size is configurable from module settings.

### Client Declaration

Client must accept an anti-abuse declaration before submission.

Submission is blocked until the declaration is accepted.

## Approval Rules

PTR / Port 25 should not be approved unless:

- WHMCS client account is active
- Selected service is active
- Invoice is not overdue
- IP belongs to the selected service
- KYC documents are uploaded
- Client details are verified
- Requested PTR hostname is valid
- PTR hostname matches a domain controlled by the client
- Mail usage reason is valid
- Client has accepted the anti-spam policy
- No relevant abuse history exists

## Risk Level System

Each request should have a risk level.

### Low Risk

- Existing verified client
- Active paid service
- Company KYC provided
- Proper mail domain
- Valid SPF/DKIM/DMARC
- Transactional/business mail only

### Medium Risk

- New client
- Individual KYC only
- Low email volume
- Partial DNS setup
- No previous abuse history

### High Risk

- New account requesting Port 25 immediately
- Incomplete KYC
- Suspicious domain
- High email volume
- Bulk email use case
- Previous abuse reports
- Mismatch between client, domain, and IP

High-risk requests should require senior admin approval.

## Workflow

1. Client opens the request page.
2. Client selects an active service and request type.
3. Client submits PTR hostname, mail domain, usage reason, mail volume, DNS status, and KYC documents.
4. System validates login, service ownership, active service status, IP format, PTR hostname format, KYC upload, and declaration acceptance.
5. Request is created with `Submitted` status.
6. Staff reviews client profile, KYC documents, service details, IP address, mail use case, DNS readiness, and abuse risk.
7. Admin approves, rejects, or requests more information.
8. Approved requests can be completed operationally and marked as `Enabled`.
9. Every action is logged with actor, action, status change, notes, and timestamp.

## Security Requirements

- KYC documents must not be publicly accessible.
- Files should be stored outside the public web root where possible.
- Only permitted admins can download KYC documents.
- File upload must validate MIME type and extension.
- Executable files must be rejected.
- Input must be sanitized.
- WHMCS CSRF token validation must be used.
- WHMCS admin permission checks must be used.
- Every approval, rejection, download, and status change must be logged.
