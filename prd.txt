# PRD: WHMCS PTR / rDNS & Outgoing Port 25 KYC Moderation Module

## 1. Product Name

**Bisup WHMCS PTR & Port 25 Approval Module**

## 2. Module Type

WHMCS Addon Module

The module will run inside WHMCS and provide:

* Client-side request form
* Admin-side moderation panel
* KYC document upload
* PTR / rDNS request management
* Outgoing Port 25 approval management
* Staff approval workflow
* Audit logs
* Ticket/email notifications

## 3. Main Goal

To create a controlled approval system inside WHMCS before enabling:

* PTR / rDNS records
* Outgoing SMTP Port 25
* Mail server sending permission

This module will reduce spam abuse, protect Bisup IP reputation, and ensure every request has proper KYC, moderation, and audit history.

## 4. Why This Module Is Needed

Currently, PTR and Port 25 approvals may happen manually through staff chat or support tickets.

That is weak because:

* No structured approval workflow
* No mandatory KYC check
* No audit log
* No abuse-risk scoring
* No staff accountability
* No proper link between client, service, IP, and approval
* Higher chance of IP blacklist issues

This module will make PTR and Port 25 approval mandatory, trackable, and controlled.

## 5. Target Users

### Client

WHMCS clients who need PTR / rDNS or outgoing Port 25 enabled for their VPS, dedicated server, or mail server.

### Support Staff

Staff who review submitted requests and verify client details.

### Admin

Senior staff who can approve, reject, enable, suspend, or flag requests.

## 6. Client Area Features

### 6.1 Client Request Page

A new page inside WHMCS client area:

**Client Area → Services → PTR / Port 25 Request**

Client can submit request for:

* PTR / rDNS only
* Outgoing Port 25 only
* Both PTR and Port 25

### 6.2 Service Selection

Client must select an active WHMCS service.

The module should only show eligible active services such as:

* VPS
* Dedicated Server
* Cloud Server
* Email Server
* Custom server products selected by admin

The system should not allow requests for cancelled, suspended, terminated, or unpaid services.

### 6.3 Required Client Fields

Client must provide:

* Service ID
* Server IP address
* Requested PTR hostname
* Mail sending domain
* Reason for request
* Type of email usage
* Estimated daily email volume
* Mail server software
* Business/organization name
* Website URL
* Contact person name
* Contact number

### 6.4 Mail Usage Type

Client must select one:

* Transactional emails
* Business emails
* Hosting client emails
* Application/system emails
* Newsletter with consent
* Other

Bulk unsolicited email, spam, phishing, spoofing, or illegal email usage must be rejected.

### 6.5 DNS Readiness Fields

Client must confirm:

* Domain has valid A record
* SPF is configured
* DKIM is configured or planned
* DMARC is configured or planned
* Mail hostname is valid
* PTR hostname belongs to the client’s domain

### 6.6 KYC Upload

Client must upload KYC documents before submitting.

For individual clients:

* Citizenship / National ID / Passport
* Address proof if required

For company clients:

* Company registration certificate
* PAN/VAT document if available
* Authorized person ID

Supported file types:

* PDF
* JPG
* PNG
* JPEG

Maximum file size should be configurable from module settings.

### 6.7 Client Declaration

Client must agree to this checkbox before submission:

> I confirm that this server will not be used for spam, phishing, spoofing, malware, bulk unsolicited emails, or illegal activity. I understand that Bisup may suspend the service, block Port 25, or terminate service if abuse is detected.

Submission should not be possible without this agreement.

## 9. Approval Rules

PTR / Port 25 should not be approved unless:

* WHMCS client account is active
* Selected service is active
* Invoice is not overdue
* IP belongs to the selected service
* KYC documents are uploaded
* Client details are verified
* Requested PTR hostname is valid
* PTR hostname matches a domain controlled by the client
* Mail usage reason is valid
* Client has accepted anti-spam policy
* No previous abuse history exists

## 10. Risk Level System

Each request should have a risk level.

### Low Risk

* Existing verified client
* Active paid service
* Company KYC provided
* Proper mail domain
* Valid SPF/DKIM/DMARC
* Transactional/business mail only

### Medium Risk

* New client
* Individual KYC only
* Low email volume
* Partial DNS setup
* No previous abuse history

### High Risk

* New account requesting Port 25 immediately
* Incomplete KYC
* Suspicious domain
* High email volume
* Bulk email use case
* Previous abuse reports
* Mismatch between client, domain, and IP

High-risk requests should require senior admin approval.

## 15. Workflow

### Step 1: Client Opens Request Page

Client selects an active service and request type.

### Step 2: Client Submits Details

Client submits PTR hostname, mail domain, usage reason, mail volume, DNS status, and KYC documents.

### Step 3: System Validation

System checks:

* Client is logged in
* Service belongs to client
* Service is active
* IP address format is valid
* PTR hostname format is valid
* Required KYC uploaded
* Declaration accepted

### Step 4: Request Created

Request status becomes:

**Submitted**

Staff/admin receives notification.

### Step 5: Staff Review

Staff checks:

* Client profile
* KYC documents
* Service details
* IP address
* Mail use case
* DNS readiness
* Abuse risk

### Step 6: Admin Decision

Admin can:

* Approve
* Reject
* Request more information

### Step 7: Technical Enablement

After approval, technical team manually creates PTR record or enables outgoing Port 25.

Status becomes:

**Enabled**

### Step 8: Audit Log Stored

Every action is logged with:

* Admin ID
* Client ID
* Action
* Status change
* Notes
* Date/time

## 16. Security Requirements

* KYC documents must not be publicly accessible
* Files must be stored outside public web root if possible
* Only permitted admins can download KYC
* File upload must validate MIME type and extension
* Reject executable files
* Sanitize all input
* Use WHMCS CSRF token validation
* Use WHMCS admin permission checks
* Log every approval/rejection/download action

## 19. Success Metrics

The module is successful if:

* No Port 25 approval happens without KYC
* No PTR request is approved without audit history
* Staff can see all pending requests clearly
* Every request is linked to a WHMCS client and service
* Abuse investigation becomes faster
* Bisup IP blacklist risk decreases
* Client verification becomes standardized

## 20. Final Recommendation

This should be developed as a WHMCS Addon Module first.

Automation for PTR creation and Port 25 firewall changes should come later. The first version should focus on moderation, KYC, approval workflow, staff accountability, and audit logs.

No Bisup staff should create PTR records or enable outgoing Port 25 outside this WHMCS approval module.
